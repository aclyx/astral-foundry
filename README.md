# Astral Foundry

Astral Foundry is a deliberate-practice Python workspace for an experienced engineer who already knows Python fundamentals, has been writing mostly TypeScript recently, and wants to become fluent in modern Python development with Astral tooling and contemporary packaging, testing, and service conventions.

This is not a toy tutorial repo. It is structured like a serious internal engineering workspace: a shared typed core library, a CLI, a FastAPI service, tests, focused examples, and docs that point you toward useful exercises instead of hiding everything behind scaffolding.

## Who This Repo Is For

- Engineers who want to refresh Python through building instead of reading disconnected snippets
- Developers who are comfortable with TypeScript and want to internalize the Python equivalents of interfaces, config, packaging, testing, and service boundaries
- People who want `uv` and `ruff` to be the default workflow, not optional extras

## Environment Setup

1. Install `uv`.

   macOS and Linux:

   ```bash
   curl -LsSf https://astral.sh/uv/install.sh | sh
   ```

   If `uv` is not available immediately afterwards, restart your shell.

2. Move into the repo and bootstrap the pinned Python environment.

   ```bash
   cd astral-foundry
   ./scripts/bootstrap.sh
   ```

3. If you prefer the manual setup flow, these are the exact commands:

   ```bash
   uv python install "$(cat .python-version)"
   uv sync --frozen
   ```

4. Optional: export environment variables from `.env.example` or create a local `.astral-foundry.toml` for CLI defaults.

   Example local config:

   ```toml
   [cli]
   api_base_url = "http://127.0.0.1:8000"
   default_output = "json"
   timeout_seconds = 5.0
   ```

`.env.example` is a reference file only. The repo does not auto-load `.env`; export variables in your shell when you want them.

## Repository Overview

```text
astral-foundry/
  pyproject.toml          # virtual workspace root, shared tooling config
  uv.lock                 # pinned dependency graph generated by uv
  packages/
    core/                # typed domain models, services, protocols, errors, config
    cli/                 # argparse-based CLI that reuses core
    api/                 # FastAPI app that reuses core
  examples/              # small runnable pattern examples
  docs/                  # learning path, architecture notes, ADRs
  scripts/               # bootstrap and full verification helpers
  .github/workflows/     # CI using uv + ruff + pytest
```

## How The Packages Fit Together

- `packages/core` is the center of the repo. It owns the `Issue`, `DigestRequest`, domain errors, protocol boundary, and the reusable issue digest service.
- `packages/cli` depends on `core` and gives you a local command-line interface plus an HTTP-backed fetch path with `httpx`.
- `packages/api` depends on `core` and exposes the same domain behavior over FastAPI routes.
- The root project is a virtual workspace orchestrator. If you come from TypeScript, think monorepo root, but with Python packaging declared in `pyproject.toml` instead of `package.json` plus path aliases.

## Daily Commands

Initial sync after pulling changes:

```bash
uv sync
```

Format the repo:

```bash
uv run ruff format .
```

Lint the repo:

```bash
uv run ruff check .
```

Run the full test suite with coverage:

```bash
uv run pytest
```

Run one test file:

```bash
uv run pytest packages/core/tests/test_services.py
```

Run the full verification sequence:

```bash
./scripts/check.sh
```

Run the CLI locally:

```bash
uv run issue-digest health
uv run issue-digest items list --limit 3
uv run issue-digest items show ISS-101
```

Run the FastAPI service locally:

```bash
uv run uvicorn api.main:app --reload
```

Call the API through the CLI:

```bash
uv run issue-digest fetch --status open --limit 2 --output json
uv run issue-digest health --remote
```

Run the standalone examples:

```bash
uv run python examples/sync_script.py
uv run python examples/async_script.py
```

## Learning Goals

- Build fluency with `uv` as the default tool for Python versioning, locking, syncing, and command execution
- Use `ruff` as both linter and formatter without adding noisy style rules
- Get comfortable with `src/` layouts, multi-package workspaces, and Python package metadata
- Practice typed domain modeling with `dataclass`, `StrEnum`, and `Protocol`
- Reuse business logic across CLI and API surfaces without framework-heavy abstractions
- Learn where async is worth using and where a synchronous function is the simpler choice
- Develop a stronger feel for pragmatic tests, explicit errors, and lightweight configuration
